# Boardgames LLM Wiki

이 저장소는 보드게임 규칙, 전략, 컴포넌트, 출처 자료를 LLM이 지속적으로 정리하고 갱신하는 Obsidian 기반 개인 위키다. 사용자는 소스와 질문을 제공하고, LLM은 위키의 구조화, 요약, 교차참조, 모순 점검, 로그 기록을 담당한다.

## 역할

- 사용자는 원본 자료를 제공하고, 어떤 게임이나 주제를 파고들지 결정한다.
- LLM은 `raw/`의 원본 자료를 읽고 `wiki/`의 마크다운 페이지를 생성하거나 갱신한다.
- 원본 자료는 수정하지 않는다. 해석, 요약, 정리, 비교, 질문의 답변은 모두 위키에 반영한다.
- 불확실한 내용은 확정적으로 쓰지 말고 `불확실`, `확인 필요`, `출처 필요`로 표시한다.
- 규칙 해석은 특히 정확성이 중요하므로, 공식 룰북과 공식 FAQ/에라타를 최우선 출처로 삼는다.

## 기본 디렉터리 구조

```text
.
├── AGENTS.md
├── raw/
│   ├── rules/
│   ├── articles/
│   ├── images/
│   └── notes/
├── wiki/
│   ├── games/
│   ├── rules/
│   ├── mechanisms/
│   ├── designers/
│   ├── publishers/
│   ├── strategy/
│   ├── sessions/
│   └── sources/
├── index.md
└── log.md
```

디렉터리가 없으면 필요한 시점에 생성한다. Obsidian 링크는 가능한 한 `[[페이지명]]` 형식을 사용한다.

## 레이어

### `raw/`

원본 자료 보관소다. 공식 룰북 PDF, 플레이어 aid, FAQ, 에라타, BGG 글, 리뷰, 전략 글, 직접 작성한 플레이 기록, 이미지 등을 넣는다.

- LLM은 `raw/` 파일을 읽기만 한다.
- 파일명은 가능하면 `게임명 - 자료명 - 날짜.md` 또는 원본 파일명을 유지한다.
- 웹에서 가져온 글은 출처 URL, 작성자, 게시일, 접근일을 문서 상단에 남긴다.
- 이미지가 규칙 설명에 중요하면 `raw/images/`에 보관하고 관련 위키 페이지에서 참조한다.

### `wiki/`

LLM이 관리하는 지식베이스다. 게임별 규칙 요약, 개념 설명, 전략 노트, 출처 요약, 비교표, 플레이 기록 종합 등을 저장한다.

- 사용자는 읽고 검토할 수 있지만, 구조 유지와 갱신은 LLM이 맡는다.
- 새 소스를 ingest할 때 기존 페이지를 찾아 갱신한다.
- 새 정보가 기존 내용과 충돌하면 조용히 덮어쓰지 말고 `모순/변경점` 섹션에 남긴다.
- 출처 없는 중요한 규칙 주장은 만들지 않는다.

### `index.md`

위키 전체의 콘텐츠 카탈로그다. LLM은 작업을 시작할 때 먼저 `index.md`를 읽어 관련 페이지를 찾는다.

각 항목은 다음 형식을 권장한다.

```markdown
- [[페이지명]] - 한 줄 요약. 출처 수: N. 최근 갱신: YYYY-MM-DD.
```

카테고리 예시는 다음과 같다.

- 게임
- 규칙/룰 해석
- 메커니즘
- 디자이너
- 출판사
- 전략
- 세션/플레이 기록
- 소스 요약
- 비교/분석

### `log.md`

작업 이력을 시간순으로 기록하는 append-only 로그다. 기존 항목은 특별한 이유 없이 수정하지 않는다.

각 항목은 다음 형식을 사용한다.

```markdown
## [YYYY-MM-DD] type | 제목

- 작업: ingest/query/lint/update
- 입력: 원본 파일 또는 질문
- 변경: 수정한 주요 페이지
- 후속: 확인할 점 또는 다음 작업
```

예시:

```markdown
## [2026-04-28] ingest | Brass: Birmingham 룰북

- 작업: ingest
- 입력: raw/rules/Brass Birmingham Rulebook.pdf
- 변경: [[Brass Birmingham]], [[네트워크 연결]], [[석탄]], [[맥주]]
- 후속: 공식 FAQ와 에라타 확인 필요
```

## Ingest 워크플로

새 원본 자료를 처리할 때는 다음 순서로 작업한다.

1. `index.md`와 `log.md`를 읽어 기존 구조와 최근 작업을 파악한다.
2. 원본 파일을 읽고 출처 정보를 확인한다.
3. 자료 유형을 분류한다: 공식 룰북, FAQ/에라타, 리뷰, 전략 글, 플레이 기록, 개인 메모, 이미지.
4. 핵심 내용, 새 정보, 기존 정보와의 충돌 가능성을 메모한다.
5. BGG, 보드라이프 등 국내외 커뮤니티/DB 사이트에서 기본 정보를 검증한다.
6. 필요한 위키 페이지를 생성하거나 갱신한다.
7. 관련 페이지 사이에 Obsidian 링크를 추가한다.
8. `index.md`를 갱신한다.
9. `log.md`에 ingest 항목을 append한다.
10. 사용자에게 변경한 페이지, 중요한 해석, 확인이 필요한 점을 짧게 보고한다.

공식 룰북과 비공식 해석이 충돌하면 공식 룰북을 우선하되, 공식 FAQ/에라타가 더 최신이면 FAQ/에라타를 우선한다. BGG 포럼, Reddit, 블로그는 비공식 출처로 표시한다.

## 기본 정보 필수 검증

룰 페이지에는 반드시 아래 기본 정보를 포함한다.

- 최소인원
- 최대인원
- 추천인원
- 메커니즘
- 난이도
- 플레이 시간
- 평점

이 기본 정보는 추측하지 않는다. 로컬 룰북 자료가 있더라도 BoardGameGeek, 보드라이프 등 국내외 커뮤니티/DB 사이트에서 반드시 확인한 뒤 작성한다.

BGG와 보드라이프의 값이 다르면 임의로 하나로 합치지 말고 출처별 차이를 함께 적는다. 예: `BGG: 3-4인 베스트 / 보드라이프: 4인 베스트, 2인 추천`, `BGG 평점: 7.8 / 보드라이프 평점: 8.1`.

보드라이프 또는 BGG에서 특정 값이 검색되지 않으면 빈칸을 추측으로 채우지 않는다. 대신 `확인 못함` 또는 `출처에서 확인되지 않음`이라고 적고, 어떤 출처에서 확인했는지 남긴다.

## 룰 설명 원칙

- 새 룰북은 가능하면 `표준 출력 형식`의 섹션 구조를 유지한다.
- 세부 규칙보다 게임의 큰 흐름을 먼저 설명한다.
- "자원을 모아서 -> 행동을 하고 -> 점수를 얻는다"처럼 플레이 감각이 잡히는 구조를 우선한다.
- 턴, 라운드, 페이즈, 종료 조건을 단계별로 나눈다.
- 자주 헷갈리는 규칙과 예외 상황은 별도 섹션으로 분리한다.
- 초보자 팁은 실제 첫 플레이에서 실수를 줄이는 방향으로 작성한다.
- 불확실한 내용은 단정하지 말고 `판본/룰북에 따라 다를 수 있음`이라고 명시한다.

## Query 워크플로

질문에 답할 때는 raw 자료를 바로 뒤지기 전에 위키를 먼저 사용한다.

1. `index.md`에서 관련 페이지를 찾는다.
2. 관련 위키 페이지를 읽는다.
3. 답변에 필요한 경우 원본 자료를 확인한다.
4. 위키와 raw 자료에 답변할 지식이 부족하면 `외부 검색 워크플로`를 따른다.
5. 답변에는 가능한 한 관련 위키 페이지와 출처를 명시한다.
6. 답변이 새 분석, 비교, 규칙 정리, 전략 가설로 가치가 있으면 사용자에게 말하고 위키 페이지로 저장한다.

룰 질문에는 다음을 구분해서 답한다.

- 확정 규칙: 공식 룰북/FAQ로 확인된 내용
- 해석: 공식 문구를 바탕으로 한 추론
- 하우스룰/취향: 공식 규칙이 아닌 플레이 그룹 선택
- 확인 필요: 출처가 부족하거나 충돌하는 내용

## 외부 검색 워크플로

프로젝트 안에 충분한 지식이 없거나, 기존 지식이 오래되었거나, 공식 FAQ/에라타 확인이 필요한 경우 외부 검색으로 위키를 보강한다.

외부 검색을 사용하는 경우:

- `index.md`, 관련 `wiki/` 페이지, `raw/` 자료를 확인했지만 답변할 근거가 부족할 때
- 특정 게임의 공식 룰북, 공식 FAQ, 에라타, 디자이너 답변, 퍼블리셔 공지를 찾아야 할 때
- 게임 기본 정보, 디자이너, 출판사, 발매 연도, 인원수, 플레이타임, 평점 같은 메타데이터가 비어 있을 때
- 기존 페이지에 `needs-source`, `확인 필요`, `출처 필요`가 남아 있을 때
- 사용자 질문이 최신 정보나 현재 판매판/개정판 차이를 요구할 때

검색 우선순위:

1. 공식 퍼블리셔/제작사 사이트
2. 공식 룰북 PDF, 공식 FAQ, 공식 에라타
3. BoardGameGeek(`https://boardgamegeek.com/`) 게임 페이지, 파일, 포럼의 디자이너/퍼블리셔 답변
4. 디자이너 또는 퍼블리셔의 공식 SNS/뉴스레터/블로그
5. 보드라이프(`https://boardlife.co.kr/`)의 한글 룰 설명, 후기, 질문/답변, 자료
6. 신뢰할 수 있는 리뷰, 전략 글, 커뮤니티 토론

BoardGameGeek와 보드라이프에 충분한 정보가 없으면 다른 커뮤니티, 블로그, Reddit, YouTube 설명, 퍼블리셔 관련 뉴스 등을 추가로 확인해도 된다. 단, 이 경우 공식 출처가 아닌 커뮤니티/개인 해석임을 명확히 표시한다.

검색 후 처리 절차:

1. 검색한 쿼리와 확인한 주요 URL을 메모한다.
2. 공식/비공식 출처를 구분한다.
3. 보존 가치가 있는 웹 자료는 `raw/articles/` 또는 `raw/rules/`에 마크다운 요약으로 남긴다. 파일명에는 게임명, 자료명, 접근일을 포함한다.
4. 원본 링크, 작성자, 게시일, 접근일을 출처 요약 페이지와 관련 위키 페이지에 기록한다.
5. 새 정보로 `wiki/`의 게임, 규칙, 메커니즘, 전략, 출처 페이지를 갱신한다.
6. 기존 내용과 충돌하면 `모순/변경점` 섹션에 무엇이 바뀌었는지 남긴다.
7. `index.md`와 `log.md`를 갱신한다.

외부 검색으로 얻은 내용은 바로 확정 지식으로 섞지 않는다. 공식 출처가 아니면 `비공식`, `커뮤니티 해석`, `전략 의견`처럼 성격을 표시한다.

## Lint 워크플로

주기적으로 위키 건강검진을 수행한다.

확인할 항목:

- `index.md`에 없는 위키 페이지
- 존재하지 않는 링크
- 중요한 개념인데 별도 페이지가 없는 항목
- 같은 규칙을 다르게 설명하는 페이지
- 공식 FAQ/에라타로 업데이트가 필요한 낡은 설명
- 출처 없는 중요한 주장
- 고립된 페이지
- 너무 긴 페이지에서 분리할 만한 하위 주제
- 외부 검색으로 보강할 만한 정보 공백

Lint 결과는 필요하면 `wiki/maintenance/` 아래에 저장하거나 `log.md`에 요약한다.

## 출처와 인용 규칙

- 규칙 설명에는 가능하면 원본 파일명, 섹션명, 페이지 번호를 남긴다.
- 웹 출처는 URL과 접근일을 남긴다.
- 직접 플레이 경험은 출처가 아니라 `경험 메모`로 표시한다.
- 모르는 정보를 추측해 채우지 않는다.
- 위키 안에 지식이 없거나 출처가 부족하면 `외부 검색 워크플로`에 따라 검색하고, 검색 결과를 위키와 로그에 반영한다.
- 외부 검색으로 찾은 내용은 출처의 성격과 신뢰도를 함께 남긴다.

## 네이밍 규칙

- 파일명은 사람이 읽기 쉬운 한국어 또는 공식 영문명을 사용한다.
- 게임명은 가능하면 BGG/박스 표기와 일치시킨다.
- 같은 게임의 한국어명과 영문명이 모두 쓰이면 게임 페이지의 `aliases`에 넣는다.
- 페이지 제목과 파일명은 가능하면 일치시킨다.
- 특수문자 `/`, `:`, `?`, `*`는 파일명에서 피한다.

## 태그 규칙

권장 태그:

- `boardgame`
- `game`
- `rule`
- `faq`
- `errata`
- `mechanism`
- `strategy`
- `session`
- `source`
- `designer`
- `publisher`
- `needs-source`
- `needs-review`

태그는 과하게 늘리지 않는다. Obsidian 탐색과 Dataview 조회에 실제로 쓸 태그만 유지한다.

## 작업 원칙

- 작은 변경이라도 관련 `index.md`와 `log.md`를 함께 갱신한다.
- 사용자가 특정 게임에 대해 자료를 넣으면 게임 페이지를 중심 허브로 만든다.
- 규칙 페이지는 플레이 중 바로 찾을 수 있게 간결하고 예시 중심으로 쓴다.
- 전략 페이지는 확신도를 표시하고, 검증되지 않은 가설은 가설로 남긴다.
- 같은 내용을 여러 페이지에 복사하기보다 대표 페이지를 만들고 링크한다.
- Obsidian에서 읽기 쉬운 마크다운을 우선한다. 복잡한 HTML은 쓰지 않는다.
